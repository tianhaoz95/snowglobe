# Design: Cacheback Speculative Decoding

## 1. Overview
This document outlines the implementation of **Cacheback Decoding** in the Snowglobe engine. Cacheback is a training-free, model-agnostic speculative decoding method that leverages n-gram cache tables to generate draft sequences. It exploits the "burstiness" and locality of language to predict future tokens with extremely low overhead.

## 2. Core Concepts (from Cacheback Paper)
- **Minimalist Drafter**: Uses a simple LRU cache instead of a secondary neural draft model.
- **n-gram Mapping**: Maps a *leader* (tuple of tokens) to multiple *followers* (tuples of tokens).
- **Recursive Drafting**: Drafts are generated by recursively querying the cache using the tail tokens of the current sequence.
- **Dual-Table Strategy**:
    - **Dynamic Table**: Populated during the current generation and initialized from the prompt.
    - **Frozen Table**: Optional pre-built table from large corpora to mitigate cold starts.

## 3. Proposed Implementation in Snowglobe

### 3.1. Cache Table Structure
In Rust, the cache will be implemented using a hash map combined with an LRU eviction policy.

```rust
struct CacheTable {
    // Maps a leader (e.g., 1 token) to a list of followers (e.g., 3 tokens each)
    entries: LruCache<Vec<u32>, Vec<Vec<u32>>>,
    leader_len: usize,   // LL (e.g., 1)
    follower_len: usize, // FL (e.g., 3)
}
```

### 3.2. Drafting Logic
The `SpeculativeRunner` will maintain this `CacheTable`. During the `execute` call:
1.  **Lookup**: Take the last `leader_len` tokens from the current session sequence.
2.  **Generate**: Retrieve the most recent follower from the cache.
3.  **Recurse**: Append the follower to the sequence and repeat until the speculation limit `k` is reached or a cache miss occurs.

### 3.3. Update Logic
After the target model validates the speculated tokens:
1.  **Observation**: Identify all new leader-follower pairs in the *accepted* sequence.
2.  **Insertion**: Update the `CacheTable` with these new pairs, moving them to the front of the LRU.

### 3.4. Integration with ModelRunner V2
Cacheback fits naturally into the `ModelRunner` V2 interface:
- **`execute`**: Performs the cache-based drafting, then calls `target.execute` with the concatenated tokens.
- **`truncate_cache` / `fork_state`**: No specialized state needs to be managed for these beyond delegating to the target model, as the cache is separate from the transformer state.

## 4. Performance Expectations
- **Drafting Latency**: Microseconds (compared to milliseconds for small neural drafters).
- **Memory Overhead**: Minimal (a few MBs for thousands of entries).
- **Speedup**: Significant in repetitive or localized text generation (e.g., code, structured data, or formal documents).

## 5. Implementation Steps
1.  **Drafting Cache**: Implement an LRU-based n-gram cache in `engine/src/model/speculative.rs`.
2.  **Drafting Loop**: Implement the recursive lookup logic.
3.  **Validation Integration**: Ensure the `SpeculativeRunner` correctly identifies accepted tokens and updates the cache.
4.  **Prompt Initialization**: Populate the cache from the prompt during the first `execute` (prefill) call.
